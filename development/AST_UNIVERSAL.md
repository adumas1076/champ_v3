# AST — Anthony's Agentic Solution Thinking
**Universal Creator Methodology**

---

> **Dr. Frankenstein to create. ISF + Autoresearch to fix. Lateral thinking to see what others miss. Multi-session to move at 3x speed. Never build from scratch. Never guess twice. Never bottleneck.**

---

## Two Modes

### MODE 1: CREATE (Building something new)

```
WANT TO BUILD X?
    │
    ├── What already exists? (references, repos, frameworks, specialists)
    │
    ├── Dr. Frankenstein: stitch working parts, build the 20% gap
    │
    ├── Multi-session: split across parallel sessions, each owns its files
    │
    └── Ship it
```

### MODE 2: FIX (Something broke)

```
X ISN'T WORKING. Y IS HAPPENING.
    │
    ├── ISF: Diagnose correctly (understand system → root cause)
    │
    ├── Lateral thinking: where is this already solved?
    │
    ├── Autoresearch: generate solutions → score → test → keep/discard → loop
    │
    ├── Dr. Frankenstein: solution might exist in another build — stitch it
    │
    └── Extract the lesson. Store it. Next time starts better.
```

## The Components

| Component | What It Does |
|---|---|
| **Dr. Frankenstein** | Never build from scratch. 80% exists somewhere. Stitch proven parts. Build the gap. |
| **ISF** | Diagnose problems correctly. Understand → Identify → Root Cause. (See ISF_UNIVERSAL.md) |
| **Autoresearch** | Self-improving solution loop. Generate → Score → Test → Keep/Discard → Repeat. |
| **Lateral Thinking (de Bono)** | Think sideways. Solutions from ANY domain — frameworks, ideologies, industries, principles. |
| **Systems Thinking (Meadows)** | See the whole system, not parts. Inputs, outputs, feedback loops. |
| **Principles-Based (Dalio)** | Create rules, test them, improve them. Let principles decide, not emotions. |
| **Multi-Session** | Parallelize across sessions. Each session owns its files. No two sessions edit the same file. |

## 7 Rules

1. **Never build from scratch.** Find what exists. Stitch it. Build the gap.
2. **Never guess twice.** Failed? You have new data. Use it.
3. **Never bottleneck.** Split work across parallel sessions. No conflicts.
4. **Never think linearly.** Solutions come from any domain. Go sideways.
5. **Never start from zero knowledge.** References, frameworks, past lessons — use them.
6. **Never fight the system.** If your solution hacks around the design, you're wrong. Use what the system provides.
7. **Always extract the lesson.** Every build, every fix produces a reusable pattern. Store it.

## How to Use in Any Build

### Starting a new project
1. What exists that does something similar? (repos, references, frameworks)
2. Dr. Frankenstein: what parts can I stitch?
3. What's the 20% gap I need to build?
4. Split across sessions — each owns a piece
5. Ship

### Something broke
1. Open ISF_UNIVERSAL.md
2. Follow the formula
3. If you're stuck at Step 8, think LATERALLY — where is this already solved in another domain?
4. Let autoresearch generate and test candidates
5. Extract the lesson

### Running multiple sessions
- Each session gets a clear scope (specific files/folders it owns)
- Sessions can READ from shared files but only WRITE to their own
- Handoff prompt includes: what to build, what files to touch, what NOT to touch
- Check other sessions' output before yours conflicts

## Multi-Session Handoff Template

```
We are building [PROJECT NAME].

YOUR TASK: [specific task]

FILES YOU OWN (only create/modify these):
- [list files]

FILES TO READ (do not modify):
- [list reference files]

CONTEXT:
- [link to architecture docs]
- [link to reference builds]

RULES:
- Do NOT modify files outside your scope
- Read reference files for consistency
- Test everything you build
- Follow ISF if you hit a blocker (see ISF_UNIVERSAL.md)
```
